# Analytics Code Style

Code written under this skill is for **controlled internal environments** — analytics teams,
data scientists, notebook runners. Inputs are known and constrained. Failures should surface
loudly and immediately, not be silently swallowed.

---

## Core Philosophy

**Happy path only.** Write for the expected input. Don't defensively code around hypotheticals.

**Throw, don't catch.** If something breaks, let it break visibly. No `try/except` wrappers
unless the user explicitly asks. Raw tracebacks are features here — they point directly at
what needs fixing.

**Comments explain *why*, not *what*.** The code already says what it does. A comment is only
worth writing if it explains a non-obvious decision, a gotcha, a business rule, or why an
alternative was rejected.

**Terse but readable.** No unnecessary abstraction. No wrapper functions for things that only
happen once. Flat is better than nested. Code reads top to bottom.

---

## Comment Rules

### Write a comment when:
- A line or block would confuse a competent reader without context
- A business rule or domain constraint drives a magic number or filter
- An approach was chosen over a more obvious alternative for a non-obvious reason
- The order of operations matters and isn't self-evident

### Never write a comment when:
- It just restates the code in English (`# loop over rows`)
- It's a section header for clearly grouped code (`# load data`, `# transform`)
- It describes standard library behavior (`# convert to datetime`)
- It was autogenerated as boilerplate

**Bad:**
```python
# Read the CSV file into a dataframe
df = pd.read_csv("sales.csv")

# Filter to only 2024 rows
df = df[df["year"] == 2024]
```

**Good:**
```python
df = pd.read_csv("sales.csv")

# Exclude partial year — 2023 data was mid-migration and has ~12% nulls in revenue
df = df[df["year"] == 2024]
```

---

## Error Handling Rules

**Do not wrap in try/except unless explicitly requested.**

Let Python, pandas, SQL, etc. raise their native errors. A `KeyError` on a missing column
is more useful than a custom message that hides the traceback.

**Bad:**
```python
try:
    result = df.groupby("region")["revenue"].sum()
except Exception as e:
    print(f"Aggregation failed: {e}")
    result = pd.Series()
```

**Good:**
```python
result = df.groupby("region")["revenue"].sum()
```

**Acceptable guard** (only when the condition is part of the business logic, not defensive coding):
```python
assert df["user_id"].is_unique, "upstream dedup step failed — check ingestion job"
```

Use `assert` for invariants you'd want to know about immediately. One line, plain message.

---

## Structure Rules

- **No unnecessary abstraction.** Don't wrap a 3-line block in a function unless it's called
  more than once or the name adds significant clarity.
- **No classes** unless the problem is inherently stateful and the user asks for it.
- **Linear flow** where possible. Avoid deep nesting. Break conditions early.
- **Variable names** should be descriptive but not verbose. `revenue_by_region` over `rev`,
  but not `total_aggregated_revenue_grouped_by_region_for_year_2024`.
- **Imports at top**, grouped: stdlib → third-party → local. No inline imports.
- **Magic numbers** get a constant at the top of the file if they appear more than once,
  or a comment explaining their origin if they appear once.

## Language-Specific Patterns

### Python / Pandas
```python
import pandas as pd
from pathlib import Path

DATA_DIR = Path("data/raw")
CUTOFF_DATE = "2024-01-01"  # aligned with fiscal year start per finance team

df = pd.read_parquet(DATA_DIR / "events.parquet")
df = df[df["event_date"] >= CUTOFF_DATE]

summary = (
    df.groupby(["region", "product"])
    .agg(revenue=("amount", "sum"), orders=("order_id", "nunique"))
    .reset_index()
)

summary.to_csv("output/summary.csv", index=False)
```

### SQL
```sql
-- attribution window is 30 days per marketing team definition
with attributed as (
    select
        e.user_id,
        e.converted_at,
        t.campaign_id,
        row_number() over (partition by e.user_id order by t.touched_at desc) as touch_rank
    from conversions e
    join touches t
        on t.user_id = e.user_id
        and t.touched_at between dateadd(day, -30, e.converted_at) and e.converted_at
)

select campaign_id, count(*) as conversions
from attributed
where touch_rank = 1
group by campaign_id
```

### PySpark
```python
from pyspark.sql import functions as F

df = spark.read.parquet("s3://bucket/events/")

# deduplicate before aggregation — source emits duplicate events on retry
df = df.dropDuplicates(["event_id"])

result = (
    df.groupBy("region")
    .agg(F.sum("amount").alias("revenue"))
)

result.write.mode("overwrite").parquet("s3://bucket/output/")
```

## What to Omit

Never add:
- Logging setup (`logging.basicConfig(...)`) unless the script is long-running or scheduled
- `argparse` / CLI flags unless the user is building a CLI tool
- `.env` / config loading unless inputs genuinely vary across environments
- `__main__` guards in notebook-style scripts
- Type hints on every function unless the user asks or the codebase already uses them
- Docstrings on private/internal functions — only on public APIs or complex transformations

---

## Checklist Before Outputting Code

1. Does every comment explain *why*, not *what*?
2. Is there any try/except that wasn't explicitly requested? Remove it.
3. Is there a function or class that only wraps a single call? Inline it.
4. Are there section header comments (`# Step 1: Load`) that add no value? Remove them.
5. Are magic numbers explained or named?
6. Does the code read top-to-bottom without jumping around?
